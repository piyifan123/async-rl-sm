# Async-Gym Design Document

## 1. Problem Statement

### Sample Scheduler for Asynchronous, Off-Policy RL with External Constraints

Data points in an off-policy RL engine move through several stages:

1. **Pending submission** — task is known but no work has started.
2. **In-flight (rollouts)** — trajectories are being generated by inference replicas.
3. **In-flight (partial rollouts + partial judging)** — some trajectories are still
   rolling out while others have already entered the judging pipeline.
4. **In-flight (judging)** — all rollouts are complete; remaining trajectories are
   being scored by judge replicas.
5. **Ready for consumption** — all trajectories are rolled out and judged; the task
   can be consumed by a trainer.
6. **Consumed** — the trainer has consumed the task.

The scheduler must respect several external constraints:

| Constraint | Description |
|---|---|
| Inference replicas | Bounds rollout throughput (finite GPU slots). |
| Training replicas | Bounds how fast ready tasks are consumed. |
| Judge replicas | Bounds judging throughput (may be variable / bursty). |
| SLA problem lengths | Some problems produce short rollouts, others very long ones; the scheduler must avoid head-of-line blocking. |

The goal is to design and implement a scheduler that models these variables and
provide a discrete-event simulation that demonstrates correct behavior over N
training steps with configurable rollout-length and judge-latency distributions.

## 2. Data Model — Task State Machine

### 2.1 Overview

Each RL **task** represents a single problem that requires `N` trajectories to be
generated and judged before it is ready for training.  `N` is a global
configuration value shared across all tasks.

A task is modelled as a **state machine** whose high-level state is *derived* from
per-trajectory counters rather than being set manually.  This keeps the source of
truth in the counters and eliminates the possibility of state/counter drift.

### 2.2 Trajectory pipeline (per-task stages)

Each trajectory for a task lives in exactly one stage at any point in time.  The
five stages form a linear pipeline:

```
pending_rollout ──▶ rolling_out ──▶ pending_judge ──▶ judging ──▶ judged
    (counter)      (InFlight list)    (counter)    (InFlight list)  (counter)
```

Three stages are plain **integer counters** (trajectories are fungible while
waiting).  The two *in-flight* stages are **lists of `InFlight` items**, each
carrying its own tick-based duration.  This enables simulation of variable
rollout lengths and judge latencies.

The **invariant** that must always hold:

```
pending_rollout + len(rolling_out) + pending_judge + len(judging) + judged == N
```

| Stage | Type | Meaning |
|---|---|---|
| `pending_rollout` | `int` | Trajectories waiting to be dispatched for rollout. |
| `rolling_out` | `list[InFlight]` | Trajectories currently being generated, each with its own tick-duration. |
| `pending_judge` | `int` | Trajectories whose rollout is complete but not yet dispatched to a judge. |
| `judging` | `list[InFlight]` | Trajectories currently being scored, each with its own tick-duration. |
| `judged` | `int` | Trajectories that have been both rolled out and judged. |

#### The `InFlight` item

```python
@dataclass
class InFlight:
    total_ticks: int       # assigned duration at dispatch time
    elapsed_ticks: int     # incremented each tick()

    remaining_ticks -> int # = total_ticks - elapsed_ticks
    done -> bool           # = elapsed_ticks >= total_ticks
```

Each `InFlight` represents one trajectory whose work takes `total_ticks`
simulation ticks to complete.  The scheduler assigns the duration at dispatch
time (e.g. sampled from a rollout-length or judge-latency distribution).

### 2.3 Derived task state

The task-level state is a pure function of the counters / list lengths:

| State | Condition (evaluated top-to-bottom, first match wins) |
|---|---|
| `CONSUMED` | Explicit flag set after trainer consumption. |
| `READY` | `judged == N` |
| `JUDGING` | `pending_rollout == 0 and rolling_out == 0` and judge pipeline has remaining work (`pending_judge > 0 or judging > 0`) |
| `PARTIAL` | Both rollout and judge pipelines have work: `(pending_rollout > 0 or rolling_out > 0)` **and** `(pending_judge > 0 or judging > 0 or judged > 0)` |
| `ROLLING_OUT` | Rollout pipeline has started but nothing has entered judge pipeline yet: `pending_rollout < N` and `pending_judge == 0 and judging == 0 and judged == 0` |
| `PENDING` | `pending_rollout == N` (nothing started) |

### 2.4 Transition interface

The scheduler drives the task forward through two mechanisms:

1. **Dispatch methods** — explicitly called by the scheduler to submit work.
2. **Tick method** — advances all in-flight items by one simulation tick;
   items that reach their duration auto-transition to the next stage.

#### Dispatch methods

| Method | From → To | Semantics |
|---|---|---|
| `submit_rollouts(durations)` | `pending_rollout → rolling_out` | Dispatches `len(durations)` trajectories to inference replicas. Each duration (in ticks) is assigned to one `InFlight` item. |
| `submit_judges(durations)` | `pending_judge → judging` | Dispatches `len(durations)` trajectories to judge replicas. Each duration is assigned to one `InFlight` item. |
| `consume()` | (sets consumed flag) | Trainer marks the task as consumed (requires `READY`). |

#### Tick-driven completion

| Method | Effect |
|---|---|
| `tick() → TickResult` | Advances every `InFlight` item in `rolling_out` and `judging` by one tick. Items whose `elapsed_ticks` reaches `total_ticks` are removed from the in-flight list and their target counter is incremented (`rolling_out → pending_judge`, `judging → judged`). Returns a `TickResult(rollouts_completed, judges_completed)`. |

Each transition validates preconditions and re-checks the invariant.

### 2.5 Simulation flow (single tick)

```
   Scheduler                          Task
      │                                 │
      │── submit_rollouts([3,5]) ──────▶│  pending_rollout -= 2
      │                                 │  rolling_out += [InFlight(3), InFlight(5)]
      │                                 │
      │── submit_judges([2]) ──────────▶│  pending_judge -= 1
      │                                 │  judging += [InFlight(2)]
      │                                 │
      │── tick() ──────────────────────▶│  for each InFlight: elapsed += 1
      │                                 │  if done: move to next stage
      │◀── TickResult(r=0, j=0) ───────│
      │                                 │
      │── tick() ──────────────────────▶│  ...
      │◀── TickResult(r=0, j=1) ───────│  judge (dur=2) completed
      │                                 │
```

## 3. Resource Pools — Modelling Replica Capacity

### 3.1 Motivation

In a real deployment, rollout throughput is bounded by the number of inference
GPU replicas, and judging throughput is bounded by the number of judge replicas.
These are **global shared resources** — all tasks compete for the same pool of
slots.  The `Task` state machine is intentionally unaware of capacity; it only
tracks its own trajectory counters.  A separate `ReplicaPool` class models the
finite capacity constraint.

### 3.2 `ReplicaPool`

A lightweight dataclass with acquire / release semantics:

```python
@dataclass
class ReplicaPool:
    name: str          # human-readable label ("inference", "judge")
    capacity: int      # total concurrent slots (>= 1)

    in_use -> int      # currently occupied slots
    available -> int   # = capacity - in_use

    acquire(n) -> None # reserve n slots   (raises if n > available)
    release(n) -> None # free n slots      (raises if n > in_use)
```

**Invariant:** `0 <= in_use <= capacity` — checked after every acquire / release.

A typical deployment has two pools:

| Pool | Models | Example capacity |
|---|---|---|
| `inference_pool` | GPU slots for trajectory generation | 8 |
| `judge_pool` | Judge API / model replicas | 4 |

### 3.3 Coordination pattern

The scheduler sits between the pools and the tasks.  Each simulation tick
follows this pattern:

```
  ┌───────────┐         ┌──────────────┐         ┌──────────┐
  │ Inference  │◀─ acquire/release ─▶│  Scheduler   │◀─ submit/tick ─▶│  Tasks   │
  │ Pool       │                     │              │                 │          │
  └───────────┘         ┌──────────────┘         └──────────┘
  ┌───────────┐         │
  │ Judge     │◀────────┘
  │ Pool      │
  └───────────┘
```

**Per-tick pseudocode:**

```python
# Phase 1: Dispatch — scheduler decides what to submit
for task in active_tasks:
    # Rollouts: respect inference capacity
    can_submit = min(task.pending_rollout, inference_pool.available)
    if can_submit > 0:
        durations = sample_rollout_durations(can_submit)
        task.submit_rollouts(durations)
        inference_pool.acquire(can_submit)

    # Judges: respect judge capacity
    can_judge = min(task.pending_judge, judge_pool.available)
    if can_judge > 0:
        durations = sample_judge_durations(can_judge)
        task.submit_judges(durations)
        judge_pool.acquire(can_judge)

# Phase 2: Tick — advance time, completions free slots
for task in active_tasks:
    result = task.tick()
    if result.rollouts_completed > 0:
        inference_pool.release(result.rollouts_completed)
    if result.judges_completed > 0:
        judge_pool.release(result.judges_completed)
```

### 3.4 Why capacity is separate from the Task

- **Global resource** — a single pool is shared by many concurrent tasks.
- **Task stays pure** — it validates its own counter invariant, not global limits.
- **Reusable** — the same `ReplicaPool` class is used for both inference and
  judge replicas (different `capacity`, same logic).
- **Testable independently** — pool logic has its own unit tests with no Task
  dependency.

## 4. Simulation Runner

### 4.1 Overview

The `Simulation` class is the top-level orchestrator.  It wires together a set
of `Task` instances and two `ReplicaPool`s (inference and judge), then runs a
synchronous discrete-event loop until every task is consumed or a safety tick
limit is reached.

All configuration is captured in a frozen `SimConfig` dataclass.  An optional
RNG seed makes runs fully reproducible.

### 4.2 Per-tick loop

Each tick executes three phases in order:

```
Phase 1 — Dispatch    greedily submit rollouts/judges for active tasks
Phase 2 — Tick        advance all in-flight items, release pool slots
Phase 3 — Consume     auto-consume any task that reached READY
```

**Phase 1 (Dispatch):** Iterates tasks in creation order.  For each
non-consumed task, dispatches as many rollouts and judges as both the task's
pending count and the corresponding pool's available slots allow:

```python
can_rollout = min(task.pending_rollout, inference_pool.available)
can_judge   = min(task.pending_judge, judge_pool.available)
```

Duration for each dispatched trajectory is sampled from a user-supplied
distribution function `(n: int, rng: Generator) -> list[int]`.

**Phase 2 (Tick):** Calls `task.tick()` on every active task.  Completed
rollouts release inference pool slots; completed judges release judge pool
slots.

**Phase 3 (Consume):** Any task in `READY` state is immediately consumed.
This phase will later be replaced by a training gate.

### 4.3 Configuration (`SimConfig`)

| Field | Type | Description |
|---|---|---|
| `n_tasks` | `int` | Total tasks to simulate (>= 1). |
| `n_trajectories` | `int` | Trajectories per task (>= 1). |
| `inference_capacity` | `int` | Inference replica pool size (>= 1). |
| `judge_capacity` | `int` | Judge replica pool size (>= 1). |
| `rollout_duration_fn` | `(int, Generator) -> list[int]` | Samples rollout durations. |
| `judge_duration_fn` | `(int, Generator) -> list[int]` | Samples judge durations. |
| `max_ticks` | `int` | Safety tick limit (default 100 000). |
| `seed` | `int \| None` | RNG seed for reproducibility. |

Two built-in duration factories are provided:

- `constant_duration(value)` — every item takes exactly *value* ticks.
- `uniform_duration(low, high)` — each item sampled uniformly from `[low, high]`.

### 4.4 Result types

`SimResult` captures:

- `ticks_elapsed` — how many ticks the simulation ran.
- `tasks_completed` — how many tasks reached `CONSUMED`.
- `history` — a `list[TickStats]` with per-tick snapshots (dispatch counts,
  completion counts, pool utilisation, task-state distribution).

### 4.5 Entry point

`run_sim.py` at the project root configures a default `SimConfig` and prints
a summary table with periodic snapshots.  Run via:

```bash
uv run run_sim.py
```

### 4.6 Design decisions

- **All tasks created at tick 0** — simplest starting point; arrival
  scheduling is a future extension.
- **Greedy dispatch order** — tasks are visited in creation order; a priority
  policy (e.g. closest-to-completion first) can be swapped in later.
- **Auto-consume** — since training is excluded, `READY` tasks are consumed
  immediately; this slot will later be replaced by a training gate.
- **No async / no threads** — the loop is pure synchronous discrete-event
  simulation.  "Async" in the project name refers to the RL pipeline model,
  not Python coroutines.
